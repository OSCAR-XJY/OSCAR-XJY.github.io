Primality Testing

Posted on 2018-04-27 | In Number-Theoretic Algorithms

Description

A prime number is a natural number greater than 1 that cannot be formed by multiplying two smaller natural numbers. A primality test is an very useful algorithm, especially in cryptography. It tests whether the input integer is prime or not. In order to test a very huge integer within a reasonable amount of time, a high-performance primality testing algorithm is worth to be studied. For simplicity, we will use $n$ to represent the number needed to be tested. $\left( {n > 1} \right)$

Naive Trial Method

If $n$ equal to $2$, then we know $n$ is a prime.
If $n$ larger than $2$, the simplest method is to check whether existed an integer $d$ ($2 \le d \le n - 1$) divides $n$.

The time complexity of this method is ${\rm O}\left( n \right)$, so this method is not so fast. It can meet our needs only when the input $n$ is truely small. However, once $n$ becomes a larger integer, this algorithm consumes a lot of time to compute the answer. For instance, we take $n = {2^{31}} - 1 = 2147483647$ as a test case. A program using this method takes about $15s$ to find the answer that $n$ is a prime in my computer. This is not tolerable in our daily application, in which the encryption and decryption should complete within an instant. Not to mention, in practice, $n$ can be a much more huge integer even contains hundreds of digits.

Root Trial Method

Since $n$ and $d$ are both positive integer, we can easily prove this:
$$
\begin{cases}
d \ge \sqrt n, &\text{$d$ is a divisor of $n$} \\
\frac{n}{d} \le \sqrt n, & \text{$\frac{n}{d}$ is another divisor of $n$}
\end{cases}
$$
So once $n$ has divisors $d$ $\left( {2 \le d \le n - 1} \right)$,
then at least one of the divisors $d$ that $d \le \sqrt n $ .
Hence we only need to test all possible divisors $d$ that $2 \le d \le \sqrt n $.

The time complexity of this method is ${\rm O}\left( {\sqrt n } \right)$. This root trial method is much more rapid than the naive trial method. If we still use the test case $n = 2147483647$, a program using this algorithm will takes only $0.004s$ to find the correct answer. But can we count on this method to deal with some even bigger integers? Next, we will use $n = 56784968830078273$ as a test case. By implementing this algorithm, a program in my computer uses about $5.6s$ to find out that $n$ is a prime, which is not as fast as we demanded. So a better method still needed to be developed.

Primorial Trial Method

We noticed the divisor $d$ can be written as:

$$6k + i,i \in \left[ { - 1,4} \right]\left( {i,k \in {\rm{Z}}} \right)$$

Since: $n$ can be divided by $d = 6k + i,i \in \left\{ {0,2,3,\left. 4 \right\}} \right.$ $\Leftrightarrow $ $n$ can be divided by $2$ or $3$.
So first we check if $n$ ($n \ne 2$ or $3$) can be divided by $2$ or $3$. If it does, then $n$ is a composite number.
Otherwise, we check whether $n$ is divisible by any $d$ of remained form $d = 6k + i,i \in \left\{ { - 1,\left. 1 \right\}} \right.$ $(5 \le d \le \sqrt n )$.

The time complexity of this method is also ${\rm O}(\sqrt n )$. However, by using this method, if the input $n$ is divisible by $2$ or $3$, we will know the $n$ is not a prime in immediately. Only when the $n$ is in the form of $6k \pm 1$, then we need to take farther actions. Meanwhile, we now use $d+=6$, which executes much more quicker than $d++$. There total $3-2+1=2$ different $i$, so even in the worst cases, this method is about $1/\frac{2}{6} = 3$ times faster than root trial method. On my computer, this method only takes about $1.6s$ to compute that $n = 56784968830078273$ is a prime.

Generalizing further, we can also write $d​$ in the form of:

$$30k + i,i \in \left[ {0,29} \right]\left( {i,k \in {\rm Z}} \right)$$

As we can see, $2$ divides $d$ when $i=0, 2, 4,…,28$ and $3$ divides $d$ when $i=0, 3, 6,…,27$ and $5$ divides $d$ when $i=0, 5, 10,…,25$. So first we check if $n$ ($n \ne 2$ or $3$ or $5$) can be divided by $2$ or $3$ or $5$. If it does, then $n$ is a composite number. Otherwise, we only need to take farther actions for those $d$ in of the form of $30k + i,i \in \left\{ {1,7,\left. {11,13,17,19,23,29} \right\}} \right.$ $(7 \le d \le \sqrt n )$.

This time we use $d+=30$, which executes even more quicker than $d+=6$. There are total $11-3+1=9$ different $i$. In the worst cases, this method is about $\frac{2}{6}/\frac{9}{ {30} } = 1.1$ times faster than the $6k + i$ method, and this is just a little improvement. On my computer, this method takes about $1.2s$ to compute that $n = 56784968830078273$ is a prime.

We noticed that ${p_2}# = 2 \times 3 = 6$, ${p_3}# = 2 \times 3 \times 5 = 30$. As we can imagine, if we use a huger coefficient for $k$, then it will runs even rapider. Since ${p_{\rm{4} } }# = {\rm{210} }$, corresponding algorithm has ${\rm{46 - 4 + 1 = 43} }$ different $i$; ${p_{\rm{5} } }# = {\rm{2310} }$, ${\rm{343 - 5 + 1 = 339} }$ different $i$.

But the improvement is not significant, meanwhile, more space may be occupancied to store those prime numbers we programer offered, so I only manipulate $6k+i$ method for now. (To be continued)
