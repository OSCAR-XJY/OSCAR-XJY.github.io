Primality Testing

Posted on 2018-04-27 | In Number-Theoretic Algorithms

Description

A prime number is a natural number greater than 1 that cannot be formed by multiplying two smaller natural numbers. A primality test is an very useful algorithm, especially in cryptography. It tests whether the input integer is prime or not. In order to test a very huge integer within a reasonable amount of time, a high-performance primality testing algorithm is worth to be studied. For simplicity, we will use $n$ to represent the number needed to be tested. $\left( {n > 1} \right)$

Naive Trial Method

If $n$ equal to $2$, then we know $n$ is a prime.
If $n$ larger than $2$, the simplest method is to check whether existed an integer $d$ ($2 \le d \le n - 1$) divides $n$.

The time complexity of this method is ${\rm O}\left( n \right)$, so this method is not so fast. It can meet our needs only when the input $n$ is truely small. However, once $n$ becomes a larger integer, this algorithm consumes a lot of time to compute the answer. For instance, we take $n = {2^{31}} - 1 = 2147483647$ as a test case. A program using this method takes about $15s$ to find the answer that $n$ is a prime in my computer. This is not tolerable in our daily application, in which the encryption and decryption should complete within an instant. Not to mention, in practice, $n$ can be a much more huge integer even contains hundreds of digits.

Root Trial Method

Since $n$ and $d$ are both positive integer, we can easily prove this:
$$
\begin{cases}
d \ge \sqrt n, &\text{$d$ is a divisor of $n$} \\
\frac{n}{d} \le \sqrt n, & \text{$\frac{n}{d}$ is another divisor of $n$}
\end{cases}
$$
So once $n$ has divisors $d$ $\left( {2 \le d \le n - 1} \right)$,
then at least one of the divisors $d$ that $d \le \sqrt n $ .
Hence we only need to test all possible divisors $d$ that $2 \le d \le \sqrt n $.

The time complexity of this method is ${\rm O}\left( {\sqrt n } \right)$. This root trial method is much more rapid than the naive trial method. If we still use the test case $n = 2147483647$, a program using this algorithm will takes only $0.004s$ to find the correct answer. But can we count on this method to deal with some even bigger integers? Next, we will use $n = 56784968830078273$ as a test case. By implementing this algorithm, a program in my computer uses about $5.4s$ to find out that $n$ is a prime, which is not as fast as we demanded. So a better method still needed to be developed.

Primorial Trial Method

Every composite number can be broken down into prime number factors, therefore we can only check whether existed a prime $d$ $\left( {2 \le d \le \sqrt n } \right)$ divides $n$. However, we need large space to to store the prime list, since we aim at very huge input $n$.

So we will use a compromised method named Primorial Trial Method to avoid generating the prime list. We noticed ${p_2}$# $= 2 \times 3 = 6$, then all $n$ can be written as:

$$n=6k + i,i \in \left[ { - 1,4} \right]\left( {i,k \in {\rm{Z} } } \right)$$

If:      $n = 2$, $3$
                                           $ \Rightarrow $ $n$ is a prime

Else if:    $n = 6k + i,i \in \left\{ {0,2,3,\left. 4 \right\} } \right.$ $\Leftrightarrow $ $n$ can be divided by $2$ or $3$
                                         $ \Rightarrow $ $n$ is a composite
                 $ \Rightarrow $ all other primes (except $2$ and $3$) are in the form of $6k + i,i \in \left\{ { - 1,\left. 1 \right\} } \right.$
                              $ \Rightarrow $ but not all $6k + i,i \in \left\{ { - 1,\left. 1 \right\} } \right.$ are primes

Else if:    $n=6k + i,i \in \left\{ { - 1,\left. 1 \right\} } \right.$
                       $ \Rightarrow $ we check if existed any prime $d$ $\left( {5 \le d \le \sqrt n } \right)$ divides $n$
                         $ \Rightarrow $ since all prime are in the form of $6k + i,i \in \left\{ { - 1,\left. 1 \right\} } \right.$
                  $ \Rightarrow $ so we check if any $d=6k + i,i \in \left\{ { - 1,\left. 1 \right\} } \right.$ divides $n$, $\left( {5 \le d \le \sqrt n } \right)$

However:    not all $d = 6k + i,i \in \left\{ { - 1,\left. 1 \right\} } \right.$ are primes
                                     $ \Rightarrow $ we will do redundant checks

The time complexity of this method is also ${\rm O}(\sqrt n )$. However, by using this method, if the input $n$ is divisible by $2$ or $3$, we will know the $n$ is not a prime in immediately. Only when the $n$ is in the form of $6k \pm 1$, then we need to take farther actions. Even though, this method only checks $d$ in the form of $6k \pm 1$. Therefore, this method is about $6/2 = 3$ times as fast as Root Trial Method. On my computer, this method only takes about $1.6s$ to compute that $n = 56784968830078273$ is a prime.

Generalizing further, we noticed ${p_3}$# $= 2 \times 3 \times 5 = 30$, then all $n$ can be written as:

$$n=30k + i,i \in \left[ {0,29} \right]\left( {i,k \in {\rm Z}} \right)$$

If:      $n = 2$, $3$, $5$
                                           $ \Rightarrow $ $n$ is a prime

Else if:    $n = 30k + i,i \notin \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$ $\Leftrightarrow $ $n$ can be divided by $2$ or $3$ or $5$
                                         $ \Rightarrow $ $n$ is a composite
       $ \Rightarrow $ all other primes (except $2$, $3$ and $5$) are in the form of $30k + i,i \in \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$
                    $ \Rightarrow $ but not all $30k + i,i \in \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$ are primes

Else if:   $n=30k + i,i \in \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$
                        $ \Rightarrow $ we check if existed any prime $d$ $\left( {7 \le d \le \sqrt n } \right)$ divides $n$
                $ \Rightarrow $ since all prime are in the form of $30k + i,i \in \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$
         $ \Rightarrow $ so we check if any $d=30k + i,i \in \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$ divides $n$, $\left( {7 \le d \le \sqrt n } \right)$

However:   not all $d=30k + i,i \in \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$ are primes
                                    $ \Rightarrow $ we will do redundant checks

The time complexity of this method is still ${\rm O}(\sqrt n )$. However, by using this method, if the input $n$ is divisible by $2$ or $3$ or $5$, we will know the $n$ is a composite in immediately. Only when the $n$ is in the form of $n=30k + i,i \in \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$, then we need to take farther actions. Even though, this method only checks those $d$ in the form of $d=30k + i,i \in \left\{ {1,7,11,13,17,19,23,\left. {29} \right\} } \right.$. Compared with Root Trial Method, this method is about $30/8 = 3.75$ times faster. On my computer, this method only takes about $1.2s$ to compute that $n = 56784968830078273$ is a prime.

As we can imagine, if we use a huger coefficient for $k$, then analogous algorithms may run even rapider. The chart below intuitively demonstrates that:

$n$ ${p_n}$# Faster times (compared with Root Trial Method)
2 6 ${\rm{6/(3 - 2 + 1) = 3} }$
3 30 ${\rm{30/(10 - 3 + 1) = 3} } {\rm{.75} }$
4 210 ${\rm{210/(46 - 4 + 1) = 4} } {\rm{.88} }$
5 2310 ${\rm{2310/(343 - 5 + 1) = 6} } {\rm{.81} }$
6 30030 ${\rm{30030/(3248 - 6 + 1) = 9} } {\rm{.26} }$
7 510510 ${\rm{510510/(42331 - 7 + 1) = 12} } {\rm{.06} }$
8 9699690 ${\rm{9699690/(646029 - 8 + 1) = 15} } {\rm{.01} }$
9 223092870 $223092870{\rm{ / ( } }12283531{ \rm{ - 9 + 1) = 18} } { \rm{.16} }$

Due to the increases of coefficient of $k$, the redundant checks caused by composite $d$ reduced. So the time consumption significantly drops. However, no matter how big the coefficient of $k$ we use, we will still encounter composite $d$, and do some redundant checks (as long as the input $n$ is big enough).

Ideally, if all the $d$ we using are prime, when $n$ is really big, Primorial Trial Method will be about $\ln \sqrt n $ times faster than Root Trial Method. Let $\pi \left( n \right)$ be the prime-counting function, we can easily prove that:

$$\mathop {\lim }\limits_{n \to \infty } \frac{ {\pi \left( n \right) } } { {\frac{n}{ {\ln n} } } } = 1$$

$$\sqrt n /\frac{ {\sqrt n } } { {\ln \sqrt n } } = \ln \sqrt n$$

However, if we use a larger test case $n = 8223372036854775893$, which is a little smaller than the maximal value of $long$ $long$ $int$ $(9223372036854775807)$ and reserved a margin to prevent overflow, these ${\rm O}(\sqrt n )$ algorithms still take too much time to generate answer. So other wiser algorithms are still waiting to be found and studied by us.
